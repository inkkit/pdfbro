# Folio v0.2 Design

**Date:** 2026-05-01
**Branch:** v0.2
**Sequence:** Shot 3 → Shot 2 → Shot 1

## Overview

Three sequential shots on a single branch. Each shot is independently
shippable. Shot 3 closes the most painful API surface gaps versus
Gotenberg. Shot 2 produces the first published Folio-vs-Gotenberg perf
report. Shot 1 raises BDD scenario coverage from 63 to 468 (100% port).

## Shot 3 — API Surface Gaps (~2.5 days)

### 3.1 `downloadFrom` field

**Field:** JSON array `[{url: string, extraHttpHeaders?: {[key: string]: string}}]`
present on every multipart route.

**Behaviour:** The server fetches each URL (with optional per-URL
headers) before the handler runs, injecting each response body as a
virtual uploaded file. File name is derived from the URL path's last
segment; content-type from the response `Content-Type` header.

**New module:** `crates/server/src/download.rs`
- `fetch_downloads(items, config) -> Result<Vec<(filename, bytes, content_type)>>`
- Called from the shared multipart parsing step before any route handler.

**New `ServerConfig` flags:**
- `--api-download-from-allow-list` — comma-separated regex patterns; URLs must match at least one. Empty = allow all.
- `--api-download-from-deny-list` — comma-separated regex patterns; URLs matching any are rejected. Checked before allow list.
- `--api-download-from-max-retry` — integer, default `3`. Exponential backoff between retries.
- `--api-disable-download-from` — bool flag; returns 400 on any request that includes `downloadFrom`.

**Validation order:** disabled check → deny list → allow list → fetch.
First matching deny rejects. Retry only on transient errors (5xx, timeout).

**Error responses:**
- `downloadFrom` present but feature disabled → 400
- URL matches deny list → 400
- URL not in allow list → 400
- Fetch fails after retries → 400 with reason

### 3.2 `omitBackground` field

**Field:** bool, default `false`. Present on all six chromium routes
(`/convert/url`, `/convert/html`, `/convert/markdown`,
`/screenshot/url`, `/screenshot/html`, `/screenshot/markdown`).

**Behaviour:** Calls `SetDefaultBackgroundColorOverride(RGBA 0,0,0,0)`
before rendering, making Chrome's default white canvas transparent.

**Validation:** `omitBackground=true` requires `printBackground=true`.
Returns 400 otherwise (matches Gotenberg's
`ErrOmitBackgroundWithoutPrintBackground`).

**Implementation:** One new CDP call in
`crates/engine/src/chromium/render.rs` (`apply_omit_background`),
called after `apply_emulated_media`. One validation in the route
parser in `crates/server/src/routes/chromium.rs`.

**Engine change:** Add `omit_background: bool` to `PdfOptions`
(default `false`) and `ScreenshotOptions`.

### 3.3 `GET /` root route

Returns a minimal HTML page listing all registered routes, matching
Gotenberg's root response shape. One handler in
`crates/server/src/routes/root.rs`, registered in `app.rs`.

### 3.4 `POST /forms/pdfengines/embed`

Dedicated font-embedding endpoint matching Gotenberg's route.

**Input:** multipart form — one PDF file + one or more font files.

**Implementation:** Shell out to ghostscript (`gs`) which is already
present in the runtime image. Command:
```
gs -dBATCH -dNOPAUSE -sDEVICE=pdfwrite -dEmbedAllFonts=true \
   -sOutputFile=<out> <input.pdf> <font1> <font2> ...
```

**Handler:** `crates/server/src/routes/pdfengines.rs` — new function
`embed_fonts`, registered at the route.

### 3.5 Encrypt permissions bitmask

**New field:** `permissions` on `POST /forms/pdfengines/encrypt`.
Accepts an array of permission strings. Default: all permissions
granted (matches Gotenberg).

**Accepted values:**
`print`, `printHighRes`, `copy`, `modifyContents`, `modifyAnnotations`,
`fillForms`, `extractForAccessibility`, `assembleDocument`.

**Implementation:** Map string array to lopdf's permission bit flags.
Add `permissions: Option<Vec<EncryptPermission>>` to the encrypt
request struct. Parse in `crates/server/src/routes/pdfengines.rs`,
pass through to `crates/engine/src/encrypt/`.

### 3.6 `--api-correlation-id-header` flag

**Flag:** `--api-correlation-id-header <name>` — when set, the server
reads `<name>` as the incoming correlation ID header and echoes it on
responses, in addition to `X-Request-Id`.

Allows Gotenberg clients that send `Gotenberg-Trace` to work without
reconfiguration (`--api-correlation-id-header Gotenberg-Trace`).

**Implementation:** Add `correlation_id_header: Option<String>` to
`ServerConfig`. In the request-ID middleware, check the configured
header first; fall back to `X-Request-Id`.

## Shot 2 — Performance Benchmarking (~2.5 days)

### 2.1 Structure

New workspace crate `bench/` (excluded from release build via
`Cargo.toml` `default-members`). Binary `bench` with subcommand `perf`.

### 2.2 Workloads

| # | Name | Description |
|---|------|-------------|
| 1 | `html-small` | Minimal HTML, no external assets |
| 2 | `html-large` | HTML with web fonts and images |
| 3 | `url-local` | Local axum fixture server (no real network) |
| 4 | `libreoffice-docx` | 50 KB `.docx` file |
| 5 | `pdfengines-merge` | 5 × 20-page PDFs merged |

Fixtures live in `bench/fixtures/`.

### 2.3 Profile

- 4 concurrent clients
- 60s warm-up discarded
- 2 min per workload
- 3 repetitions
- Reports median latency; flags CV > 15% as unstable

### 2.4 Metrics

- p50 / p95 / p99 latency
- Requests per second
- Error rate
- Peak container RSS (sampled via `docker stats --no-stream` every 2s)

### 2.5 Fairness

`docker-compose.bench.yml` runs both servers under identical
`cpus: "2"` / `mem_limit: 2g` cgroups. Folio uses the project's own
`Dockerfile`. Gotenberg uses its official image.

Pre-flight check: prints both servers' tool versions (Chrome,
LibreOffice) and aborts if they differ.

### 2.6 Output

`bench/results/<ISO-timestamp>/perf.md`:
- Summary table: workload × server × p50/p95/p99/RPS
- Peak RSS table
- CV warnings for unstable runs
- Mandatory caveats section (hardware, config, non-determinism)

### 2.7 PDF quality gate

After each workload response: verify `%PDF` header, non-empty body,
page count matches input via lopdf. Not byte-identical — timestamps
and object IDs differ by design. Any quality failure aborts the run.

## Shot 1 — BDD Parity (~10–12 days)

### 1.1 `Dockerfile.test` updates

Add to the existing test image:
- `poppler-utils` — provides `pdftotext` and `pdfinfo`
- `default-jre-headless` — required by verapdf
- verapdf CLI — pinned version, installed from official ZIP to
  `/opt/verapdf`, symlinked to `/usr/local/bin/verapdf`

If any tool is absent at runtime, affected steps emit a warning and
mark the scenario `skipped` (not `failed`). Local `cargo test` works
with partial assertions; full validation runs via `make docker-test`.

### 1.2 Missing step vocabulary

New and extended modules in `crates/server/tests/bdd/steps/`:

**`pdf.rs` extensions:**
- Page count — `pdfinfo` → parse `Pages:` line
- Text at page N — `pdftotext -f N -l N`
- Image count + embedded files — lopdf traversal
- PDF/A validity — `verapdf --format json` → parse `failedRules` ≤ tolerance

**`webhook.rs` (new):**
- Spawn in-process axum server on a random port
- Capture POST bodies into `FolioWorld`
- Used only for the 7 webhook scenarios (all tagged `@folio-skip`)

**`container.rs` additions:**
- Capture `tracing` subscriber output into `FolioWorld.logs: Vec<String>`
- Step: `the logs should contain "<substring>"`

**`http.rs` additions:**
- Concurrent POST with N clients
- GET/HEAD with header table
- Basic-auth scenario support

### 1.3 Scenario porting — 100%

Every one of Gotenberg's 468 scenarios gets a corresponding Folio
scenario in the matching `.feature` file, either:

- **Passing** — implemented and green
- **`@folio-skip`** with `# Reason: <one line>` — not applicable or
  deferred. Examples:
  - `# Reason: tests pdftk-specific output; Folio is lopdf-only`
  - `# Reason: tests Gotenberg pprof debug endpoint; not in Folio`
  - `# Reason: Folio uses batch API; see docs/migration-from-gotenberg.md`
    (all 7 webhook scenarios)

No scenario is left without one of these two outcomes.

### 1.4 Baseline tracking

After the full port, `crates/server/tests/bdd/baseline.json` records
each scenario as `pass`, `fail`, or `skip` with reason. Failing
scenarios are documented in the baseline; no GitHub issues are created
automatically.
