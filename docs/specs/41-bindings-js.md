# Spec 41 — Node bindings (`js` crate)

> napi-rs wrapper exposing `require('folio')`. Outline.

## Goal

Allow Node.js users to call into the engine:

```js
const folio = require('folio');
const engine = new folio.ChromiumEngine();
const pdf = await engine.htmlToPdf('<h1>Hi</h1>');
```

## Public surface

```ts
export class ChromiumEngine {
    constructor(opts?: BrowserConfig);
    htmlToPdf(html: string, opts?: { baseUrl?: string; options?: PdfOptions; request?: RequestContext }): Promise<Buffer>;
    urlToPdf(url: string, opts?: { options?: PdfOptions; request?: RequestContext }): Promise<Buffer>;
    markdownToPdf(md: string, opts?: { options?: PdfOptions; request?: RequestContext }): Promise<Buffer>;
    shutdown(): Promise<void>;
}

export interface PdfOptions { /* mirrors spec 10 */ }
export interface RequestContext { /* mirrors spec 11 */ }
export interface BrowserConfig { /* mirrors spec 10 */ }
```

## Behavior

- napi-rs `#[napi]` macros generate the bindings. All async methods return
  real `Promise`s using napi-rs's tokio integration; no `block_on` calls.
- `Buffer` returned via `napi::bindgen_prelude::Buffer`, zero-copy where
  possible.
- Errors mapped to `napi::Error` with a `code` carrying the
  `EngineError` variant name.

## Build

- `crates/js/Cargo.toml` already configured as `cdylib name = "folio"`.
- `package.json` at `crates/js/` for napi-rs publishing; CI matrix builds
  prebuilt binaries for darwin-x64/arm64, linux-x64/arm64, win32-x64.
- TypeScript declarations auto-generated by napi-rs.
